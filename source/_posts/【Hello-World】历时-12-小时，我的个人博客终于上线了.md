### 一、 缘起：从羡慕到行动 (Why I started)

其实搭建博客的想法源于一位技术很强的好友。偶然发现他有自己的网站时，我非常惊奇。最近在写科研文档，加上学习 Java 缺乏动力（黑马的课看了两个月才看完 SE），我意识到我需要一个外部激励机制。

这个博客的学习板块将作为我的“公开学习笔记”**，通过输出倒逼输入，记录我的 Java 学习进展和科研收获。虽然可能没人看，但对我这种有点表演型人格的人来说，应该还是会起到一点点作用。**

### 二、 技术栈：便宜才是王道(How I built it)

作为一个初学者，我选择了最经济的静态托管方案：**Hexo + GitHub Pages**。真是一分钱没花，我甚至没给自己买域名。这次搭建我是全程在 AI（Gemini 和豆包）的辅助下完成的，当然也被ai带到沟里很多次😎。

### 三、 功能落地：麻雀虽小，五脏俱全 (What I Delivered)

在 AI 的辅助下，我搭建了一个能跑的静态页面，还实现了一套完整的工作流：

**1. 个性化导航与分类**

通过修改 Hexo 的 _config.yml 配置文件，我重构了导航栏，将其划分为项目，学习，关于，和简历等板块。关于部分我加入了GitHub的热力图（虽然我还是一片空白）

**2. 自动化部署 （CICD）**

这是我最开心的部分。我通过配置 **GitHub Actions** 实现了自动化部署。当然这里也是我卡的最久的地方。

**以前：** 每次写完要在本地手动生成 (hexo g) 再推送到仓库。

**现在的实现：** 我只需要将 Markdown 源码 push 到 GitHub 的源分支，GitHub Actions 就会自动在云端触发构建脚本，生成静态网页并发布。这让我摆脱了对本地环境的依赖。

**3. 数据监控**

为了满足一点点“虚荣心”，我接入了 **Google Analytics**。虽然现在的 UV（独立访客）可能只有我自己，但能在后台看到流量数据的波动，让我对网站的运营有了最基础的感知。

**4. 隐私保护**

考虑到**简历**涉及个人隐私，我利用 Hexo 的加密插件 (hexo-blog-encrypt) 对特定页面进行了加密处理。只有输入正确密码才能查看。

### 四、 复盘：从被ai操控到把ai当做工具(What I learned)

刚开始，我对 Git 和配置文件一窍不通，完全是机械地复制粘贴 AI 的指令。这种“知其然不知其所以然”的状态在部署阶段让我吃尽了苦头——一旦报错，寸步难行。尤其是一个报错解决完以后下一个报错又开始，不管怎样一直找不到问题真正的本质原因。

在卡顿和试错中，我不得不停下来去理解底层的逻辑：

什么是 git commit 和分支？

GitHub Pages 的 origin 到底指向哪里？

AI 给出的配置为什么会失效？

盲目使用ai让我很痛苦，也让我发现ai确实本质上只是输出概率分布最大的东西，如果技术不进步，ai永远需要人来监管，而监管和修改需要对问题有足够的了解和理解，虽然降低了门槛但不代表我什么都不需要知道。这也是我搭建这个博客最大的收获：**工具只是辅助，理解原理才是核心。**

当然，知识层面上，我算是大致了解了开发一个静态博客的流程，并使用了很多的工具，hexo，git；熟悉了git的常见基本指令，理解GitHub的repository具体在做什么，如何把本地的改动add，commit，push到GitHub上。关键在于我对git算是入门了，后面再去学习和看文章我会轻松很多，而不是云里雾里。

### 五、 结语

这是我的第一篇博客，后面我会再利用ai总结一下自己整个开发过程遇到的技术问题，解决方案等等。未来，希望能在这里见证自己技术的成长。Keep coding, keep writing.
